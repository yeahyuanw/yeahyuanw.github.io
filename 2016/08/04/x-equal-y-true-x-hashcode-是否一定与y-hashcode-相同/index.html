x.equals(y) == true x.hashcode()是否一定与y.hashcode()相同

发表于 2016-08-04   |  

面试中经常会被问到的题目：

x.equal(y) == true x.hashcode()是否一定与y.hashcode()相同?

我看了网上的回答，发现并不理想，今天看java.lang的源码时又重新思考了一下这个问题

我的回答是：不一定

Object.java中关于hashCode()函数的描述是

如果两个对象x,y，存在x.equals(y) == true，则它们的hashCode一定相同
若果 x.equals(y) == false , 没有硬性规定, x.hashcode与y.hashcode一定要不同。实际上我们的做法是将x,y的内存地址转化为它们的hashcode

但需要注意的是Object类中equals方法的实现是

也就是，在Object类中equals比较的是两个对象的地址，而不是内容。

因此，这个问题我们需要分两种情况来考虑：

  1. 继承了Object类，但没有重写equals方法
    对于这种情况，如果x.equals(y) == true , 则它们的hashCode一定相同

  2. 继承了Object类，重写了equals方法
    这种情况的典型例子是String类, String类是重写了equals方法的，具体实现为：

可以知道 , String 类当中的equals方法是先判断地址是否相等 , 不等再判断内容.

因此 , 如果有 :

String x = new String("abc");
String y = new String("abc");

x.equals(y) == true ，但是 x.hashCode() != y.hashCode() .因为, x 与 y 的地址并不一样

String x = "abc";
String y = "abc";

则是 : x.equals(y) == true , 且 x.hashCode() == y.hashCode() .因为，String有一个字符串池，如果一个字符串已经出现过一次
那么在此赋值的时候，就会将句柄直接指向池中已经存在的字符串，所以它们不仅内容相同，地址也相同，那么hashCode也相同.
